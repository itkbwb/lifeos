# Life OS

Персональная система управления временем и проектами.

- `server/` — FastAPI + SQLite бэкенд (источник истины), см. `server/README.md`.
- `android/` — нативный Kotlin/Compose клиент под Android, см. `android/README.md`.

## Архитектура

Сервер (Raspberry Pi или любой Linux/Windows хост) хранит всё в SQLite и
отдаёт REST API. Android-приложение — это интерфейс: тянет `/api/dashboard`,
шлёт действия (`complete`/`skip`), адрес сервера настраивается в приложении
(экран «Настройки»), поддержки HTTPS/домена/авторизации пока нет — это
следующий этап после того, как сам клиент станет удобным для ежедневного
использования.

## Автообновление APK

APK не публикуется в Google Play. Вместо этого:

1. При пуше тега `vX.Y.Z` GitHub Actions (`.github/workflows/android-release.yml`)
   собирает подписанный release-APK и публикует его как GitHub Release.
2. Приложение при каждом запуске дёргает
   `GET /repos/itkbwb/lifeos/releases/latest`, сравнивает версию с
   установленной, и если есть новее — скачивает `.apk` и предлагает установку.
3. Все сборки подписаны одним и тем же ключом (хранится как секреты
   репозитория `ANDROID_KEYSTORE_BASE64` и т.д.), иначе Android требовал бы
   удаления старой версии перед установкой новой.

Чтобы выпустить новую версию:

```bash
git tag v0.1.1
git push origin v0.1.1
```

## Известные ограничения на этом этапе

- Экраны «Неделя» и «Проекты» из веб-версии в Android-клиенте пока не
  реализованы — сделаны только «Сейчас», «День» и «Настройки», как самое
  необходимое для ежедневного использования.
- Сервер работает по обычному HTTP в локальной сети — авторизация, HTTPS и
  постоянный внешний адрес (Cloudflare Tunnel и т.п.) идут отдельным этапом.
- Первая установка APK на телефон должна быть ручной (единожды) — после
  этого включится автообновление.
